# Task PostgreSQL Storage

This document records the operational shape for task PostgreSQL schemas.
It applies to task lifecycle storage, task artifact metadata, queue tables,
event and usage records, and task idempotency records.

## Migration Framework

Alembic is the authoritative schema migration framework for task PostgreSQL
schemas. Revisions are hand-authored PostgreSQL migrations rather than
autogenerated ORM metadata. Runtime task store modules must not apply schema
changes on hot request paths.

Task PostgreSQL revisions are stored under the task-owned Alembic script
location in `src/avalan/task/stores/pgsql_migrations/`. Runtime modules expose
programmatic helpers for applying, checking, inspecting, and stamping schema
state, but they do not own DDL constants or apply migrations automatically.

## Revision Layout

Task revisions live in a task-owned Alembic script location. Memory
PostgreSQL revisions live in `src/avalan/memory/permanent/pgsql_migrations/`
with a separate revision history. Task migrations must not import memory
modules or require pgvector.

The task migration version table is named `avalan_task_alembic_version` by
default. Deployments that use multiple PostgreSQL schemas may configure the
schema/search path explicitly; migration helpers must pass that setting to
Alembic rather than relying on process-global state.

## DDL Policy

Task schema migrations use PostgreSQL transactional DDL where PostgreSQL
supports it. Migrations are forward-only for production recovery. Downgrade
functions may be omitted or raise a clear diagnostic when a change cannot be
safely reversed.

Concurrent migration runners are fenced with PostgreSQL advisory locks. The
lock key is stable for task migrations and independent from memory migration
locks. Helpers should expose `upgrade`, `current`, `check`, and `stamp` so
operators can apply, inspect, validate, or intentionally mark schema state.

## Optional Dependencies

Runtime PostgreSQL task storage uses the `task-pgsql` extra. Migration tooling
is dependency-gated on Alembic and SQLAlchemy so base installs and direct
in-memory task usage do not install PostgreSQL, Alembic, SQLAlchemy, pgvector,
or asyncpg.

When migration dependencies are missing, CLI and SDK entry points must return
the stable feature-gate diagnostic
`dependency.task_pgsql_migrations_missing` without leaking import exceptions,
paths, DSNs, usernames, passwords, tokens, or stack traces.

## Existing Databases

Existing embedded task migration metadata and development databases do not
receive an automatic compatibility bridge. Local development databases may be
dropped and recreated. Operators that intentionally preserve an existing schema
must use the migration helper to stamp the appropriate Alembic revision after
verifying the schema manually.
